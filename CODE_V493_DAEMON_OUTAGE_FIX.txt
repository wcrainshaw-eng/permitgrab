ROOT CAUSE
==============================================================================

CLAUDE.md's ARCHITECTURE GROUND TRUTH section explicitly documents:

    There is exactly ONE Render service: `permitgrab` (Docker, Oregon).
    The Flask web server **and** the collection daemon thread run **in
    the same process**. There is no separate worker container.

    `render.yaml` declares a `permitgrab-worker` Background Worker
    service and `worker.py` exists in the repo, **but the service was
    never created in the actual Render project**.

    `WORKER_MODE` env var is **not set** on the live web service.

V481 added two WORKER_MODE-gated no-ops on the assumption that the
worker service was running:

  1. server.py line ~9080 — start_collectors() returns immediately
     when WORKER_MODE != '1'.

  2. routes/admin.py line 1050-1062 — POST /api/admin/start-collectors
     short-circuits with a "noop" response pointing at the nonexistent
     worker service.

The route file is self-contradictory: the docstring at line 1030
says "the permitgrab-worker Render service was never created" — and
ten lines later the guard tells you to "redeploy the worker service
from the Render dashboard" if collection is dead.

V471 PR4 was the original mistake that V473b reverted. V481
reintroduced the same regression with the same false premise.

PERFORMANCE WEDGE — LATER FOLLOW-UP
==============================================================================

V481's stated reason for the gates was real: "12 gthread request
handlers + 3 daemons share one Python GIL on a single vCPU and an
aggregate-query refresh starves every request." That perf concern
is still valid — but the wrong fix is to disable the daemons
entirely.

Real fix path for the perf wedge (NOT this PR — separate work):

  Option A (preferred): create permitgrab-worker on Render dashboard.
    1. Render dashboard → New → Background Worker.
    2. Use the same Docker image as permitgrab.
    3. Set WORKER_MODE=1 env var ON the worker service ONLY.
    4. Worker entry point: worker.py (already exists in repo).
    5. Re-add the WORKER_MODE guards (this time correctly: ONLY when
       the worker service exists and is running).

  Option B (interim): make the daemons less aggressive in-process.
    1. Lower secondary_loop frequency from 30s to 5min.
    2. Timebox each collection cycle to max 60s of GIL time.
    3. Defer expensive aggregate refreshes (V488 persona stats,
       proposed V492 city stats) to off-peak hours (3-5 AM ET).

For now (V493): single-process mode is the only working state.

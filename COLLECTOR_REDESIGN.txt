PERMITGRAB — COLLECTOR REDESIGN: CLEAN ARCHITECTURE
=====================================================
Date: March 27, 2026
Context: The current collector is overcomplicated and unreliable.
         337 configs exist but only 17 cities get fresh data. The
         "autonomy engine" that discovers sources isn't completing
         its job. The failure tracker permanently blocks cities.
         The site claims 850+ cities but 83% have zero data.

This doc replaces the current hodgepodge with a simple, testable,
auditable pipeline.


=====================================================
THE CORE PROBLEM
=====================================================

The current system works backwards:
  1. Seed 30k cities into a database
  2. List 850 of them on the site
  3. Hope the collector/discovery engine eventually finds data
  4. When it fails, nobody knows

The new system works forwards:
  1. Start from known data sources (state, county, city-level)
  2. TEST each source — does it return current permit data?
  3. If yes → add the city to a verified "prod list"
  4. Site only shows cities on the prod list
  5. Count comes from the prod list, not heuristics


=====================================================
PHASE 1: SOURCE INVENTORY (do this first)
=====================================================

Before touching any code, we need a clear picture of what data
sources actually exist and work. There are 3-4 database types
that cover US permit data:

  1. SOCRATA (data.cityofX.gov, data.county.gov)
     - Standardized API, easy to query
     - Many cities and counties host open data here
     - Can test with: GET /resource/{dataset_id}.json?$limit=1

  2. ARCGIS (services.arcgis.com, gis.cityofX.com)
     - Common for city/county GIS departments
     - REST API with feature layers
     - Can test with: /query?where=1=1&resultRecordCount=1

  3. CKAN (data.gov variants, some city portals)
     - Open data platform used by government agencies
     - API: /api/3/action/datastore_search?resource_id=X&limit=1

  4. CUSTOM / DIRECT APIs
     - Some cities have their own permit lookup systems
     - These require individual integration

TASK FOR CODE:
  Export the current source inventory. For each of the 337 active
  + 255 inactive CITY_REGISTRY configs and 25 BULK_SOURCES, list:
    - source_name / city
    - source_type (socrata / arcgis / ckan / custom)
    - endpoint URL
    - active/inactive status
    - last_successful_collection date (from DB)
    - total_permits_collected (COUNT from permits table)

  Format: CSV or JSON. We need this to see what we're working with
  before redesigning anything.


=====================================================
PHASE 2: TOP-DOWN SOURCE DISCOVERY
=====================================================

The discovery engine should work TOP-DOWN by geographic scope,
not bottom-up city by city. This is the key insight — check the
biggest sources first, because one hit validates many cities.

ORDER OF OPERATIONS:

  PASS 1: STATE-LEVEL SOURCES (~50 checks)
  -----------------------------------------
  For each US state, check if there's a statewide permit dataset
  on Socrata, ArcGIS, or CKAN. Many states publish consolidated
  building permit data.

  If found → TEST it (does it return recent records?) → if yes,
  extract the list of cities/counties covered → mark ALL those
  cities as "source verified."

  Example: If Texas has a statewide source covering 500 cities,
  one successful test validates 500 cities at once.

  Known state-level sources to check/verify:
    - NJ statewide (currently configured but mostly dead — WHY?)
    - TX (strong data, explains the Texas bias in working cities)
    - Check all 50 states systematically


  PASS 2: COUNTY-LEVEL SOURCES (~200-500 checks)
  -----------------------------------------------
  For the top ~200 counties by population (covers ~80% of US
  population), check if the county publishes permit data.

  Many counties have open data portals that cover all incorporated
  cities within the county. One county source = dozens of cities.

  Known county sources to check/verify:
    - Montgomery County MD (currently configured, degrading)
    - Miami-Dade County FL (currently INACTIVE — 34 cities)
    - San Diego County CA (configured, never collected)
    - Check top 200 counties systematically

  If found → TEST → extract covered cities → mark verified.


  PASS 3: CITY-LEVEL SOURCES (work down by population)
  ---------------------------------------------------
  For cities NOT already covered by state or county sources,
  check if the city has its own open data portal.

  Work in population order:
    Tier 1: Cities 500k+ population (Phoenix, San Diego, etc.)
    Tier 2: Cities 200k-500k
    Tier 3: Cities 100k-200k
    Tier 4: Cities 50k-100k
    ... and so on down the 30k list

  For each city:
    1. Check Socrata: search for "{city name} building permits"
    2. Check ArcGIS: search for city GIS portal
    3. Check CKAN: search data.gov for city datasets
    4. If found → TEST → if recent data → mark verified

  IMPORTANT: This is a FINITE task, not an ongoing background
  process. 30k cities, but most will be covered by state/county
  passes. The remaining city-level checks should complete in
  1-2 weeks of automated checking, not run forever.


=====================================================
PHASE 3: THE TEST PROTOCOL
=====================================================

Every source — state, county, or city — goes through the same
test before it's added to the prod list:

  TEST 1: REACHABLE
    Can we hit the endpoint? Does it return a 200?
    If not → log error, skip, move on.

  TEST 2: HAS PERMIT DATA
    Does the response contain building permit records?
    (Not parks data, not business licenses, not zoning — PERMITS)
    If not → log "wrong dataset type", skip.

  TEST 3: HAS RECENT DATA
    Does the dataset contain records from the last 30 days?
    If not → log "data exists but stale", flag for review.
    (Some datasets are updated quarterly — these are still
    usable but need different collection frequency.)

  TEST 4: PARSEABLE
    Can our collector parse the response into our permit schema?
    (address, description, filing_date, value, etc.)
    If not → log "needs custom parser", flag for dev work.

  TEST 5: COLLECTION WORKS
    Run a real collection: pull last 7 days of permits, insert
    into DB, verify they appear on the site.
    If successful → ADD TO PROD LIST.

A source passes when all 5 tests succeed. Anything that fails
gets logged with the specific failure point so it can be fixed.


=====================================================
PHASE 4: THE PROD LIST
=====================================================

This replaces the current heuristics-based city listing.

NEW TABLE: prod_cities
  CREATE TABLE prod_cities (
    id SERIAL PRIMARY KEY,
    city VARCHAR(100) NOT NULL,
    state VARCHAR(2) NOT NULL,
    city_slug VARCHAR(200) UNIQUE NOT NULL,
    source_type VARCHAR(20),       -- socrata/arcgis/ckan/custom
    source_id VARCHAR(200),        -- reference to the data source
    source_scope VARCHAR(20),      -- state/county/city
    verified_date TIMESTAMP,       -- when the source was last tested
    last_collection TIMESTAMP,     -- when we last successfully collected
    total_permits INTEGER DEFAULT 0,
    status VARCHAR(20) DEFAULT 'active',  -- active/paused/failed
    added_by VARCHAR(50),          -- 'discovery_engine' or 'manual'
    notes TEXT
  );

RULES:
  - A city appears on the site ONLY if it's in prod_cities
    with status = 'active'
  - The "X cities" count on the site = COUNT(*) FROM prod_cities
    WHERE status = 'active'
  - get_cities_with_data() is REPLACED by a simple query on
    prod_cities — no more heuristics, KNOWN_OK_CITIES, or
    VALID_STATES logic
  - The /cities page pulls from prod_cities
  - City pages for non-prod cities return 404 or redirect to
    a "coming soon" page (keep URLs reserved for SEO but don't
    pretend there's data)


=====================================================
PHASE 5: COLLECTION (simplified)
=====================================================

Once the prod list exists, collection is simple:

  EVERY 6 HOURS:
    FOR each city in prod_cities WHERE status = 'active':
      1. Call the appropriate collector (based on source_type)
      2. Pull permits from last 7 days
      3. Insert new permits into DB
      4. Log the run in scraper_runs table:
           source, city, timestamp, permits_found, status, error
      5. Update prod_cities.last_collection timestamp

  ERROR HANDLING:
    - If a source fails, LOG IT and CONTINUE to the next city
      (do NOT abort the entire run)
    - If a source fails 3 times in a row, set status = 'paused'
      and trigger an alert (do NOT permanently block it like
      the current failure tracker does after 10 failures)
    - A paused city gets retried once per day (not every 6 hours)
    - If a paused city succeeds → set status back to 'active'
    - If a paused city fails for 30 consecutive days → set
      status = 'failed' and alert for manual review

  REMOVE THE CURRENT FAILURE TRACKER.
    The file-based tracker that permanently skips cities after 10
    failures is the wrong approach. Replace with the paused/retry
    logic above.

=====================================================
THE DISCOVERY ENGINE FIX
=====================================================

The current "autonomy engine" is supposed to discover new sources
automatically but isn't completing the job. Here's what it should
actually do:

  INPUT: us_cities table (30k incorporated US cities, ranked by
         population)

  PROCESS:
    For each city (in population order):
      1. Skip if already in prod_cities
      2. Skip if already covered by a state/county source in
         prod_cities
      3. Check Socrata API for city permit datasets
      4. Check ArcGIS for city GIS portal
      5. Check CKAN for city datasets
      6. If found → run test protocol
      7. If test passes → add to prod_cities
      8. If test fails → log why, move on
      9. Log progress: "{city} checked, result: {found/not_found}"

  THIS IS A FINITE JOB. 30k cities. Even at 5 seconds per check
  (API call + test), that's ~42 hours of runtime. With state and
  county passes eliminating thousands of cities upfront, the
  actual city-level checks might be 10-15k. That's 14-21 hours.

  This should COMPLETE in 1-2 weeks, not run indefinitely.

  After the initial run completes, it can re-run monthly to check
  for newly-published datasets. But the initial pass is a one-time
  task with a clear finish line.

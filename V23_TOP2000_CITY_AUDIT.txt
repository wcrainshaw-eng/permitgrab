## 2. HOW TO DISCOVER DATA SOURCES FOR A CITY
================================================================================

For each city, Code should try these approaches IN ORDER:

### 2.1 SOCRATA DISCOVERY
Socrata is the most common open data platform in the US (SODA API).

1. Check the city's main open data portal:
   - Common domains:
     * data.cityof[name].org
     * data.[name]gov.com
     * [name].data.socrata.com
     * opendata.[name].gov
   - Look for URLs like: https://data.domain.com/resource/xxxx-xxxx.json

2. Use the Socrata Discovery API:
   curl -s "https://api.us.socrata.com/api/catalog/v1?q=building+permit&domains=data.cityofdallas.us" | python3 -m json.tool

3. Search for dataset names containing:
   - "building permit"
   - "construction permit"
   - "development permit"
   - "permit issued"
   - "permits approved"

4. Once you find a dataset ID (xxxx-xxxx), test it:
   curl -s "https://data.domain.com/resource/xxxx-xxxx.json?$limit=3&$order=date_field+DESC" | python3 -m json.tool


### 2.2 ARCGIS DISCOVERY
ArcGIS REST APIs are also very common, especially for government agencies.

1. Check if the city has a GIS portal (common patterns):
   - https://gis.[city].gov
   - https://maps.[city].gov
   - https://arcgis.com/home/search.html?q=[city] building permit

2. Try Google:
   site:arcgis.com "[city name]" building permit

3. Look for FeatureServer endpoints in this pattern:
   https://services[N].arcgis.com/[orgid]/ArcGIS/rest/services/[ServiceName]/FeatureServer/[layer]/query

4. Common org IDs:
   - Test with: services.arcgis.com, services1.arcgis.com, services2.arcgis.com, etc.

5. Test the endpoint:
   curl -s "https://services1.arcgis.com/orgid/ArcGIS/rest/services/Permits/FeatureServer/0/query?where=1%3D1&outFields=*&resultRecordCount=3&orderByFields=IssuedDate+DESC&f=json" | python3 -m json.tool


### 2.3 CKAN DISCOVERY
CKAN is a datastore platform (less common than Socrata but still used).

1. Check if the city runs CKAN:
   - https://data.[city].gov/api/3/action/datastore_search?resource_id=[id]
   - https://opendata.[city].gov/api/3/action/datastore_search?resource_id=[id]

2. Find datasets by browsing the web UI or using:
   curl -s "https://data.domain.com/api/3/action/package_search?q=building+permit" | python3 -m json.tool

3. Extract resource_id from the result and test:
   curl -s "https://data.domain.com/api/3/action/datastore_search?resource_id=UUID&limit=3&sort=date_field+desc" | python3 -m json.tool


### 2.4 STATE/COUNTY LEVEL FALLBACK
If the city doesn't have its own open data portal, check the COUNTY or STATE portal.

Examples:
- Many NJ cities use NJ's statewide Socrata (nj.gov/opendata)
  Use config option: "city_filter": {"field": "CITY_NAME", "value": "Newark"}
- Many TX cities use statewide or county portals
- Many CA cities use state-level Socrata

Format in config:
```python
"city_filter": {
    "field": "CITY_NAME",
    "value": "Newark"
}
```

The collector will automatically add this to the WHERE clause.


### 2.5 GOOGLE FALLBACK (Last Resort)
If you can't find a direct source, try:

search: "[city name]" "[state]" open data building permits
search: "[city name]" building permit data portal
search: site:socrata.com OR site:arcgis.com OR site:data.gov "[city]" permit

Sometimes you'll find a data.gov portal or academic institution that hosts data
for smaller cities.


### 2.6 IF NO SOURCE EXISTS
Mark the city with "# BLOCKED:" and move on. Don't spend more than 15 minutes
on a single city. If there's no public API and you can't find one after trying
all approaches above, it's BLOCKED.

### 3.4 FIELD MAP GUIDELINES

The field_map maps PermitGrab's standard fields to the SOURCE field names.

REQUIRED fields (at minimum):
  - "filing_date" → the date field (MUST be present)

STRONGLY RECOMMENDED fields (include if available):
  - "permit_number" → permit ID/number
  - "permit_type" → type of permit
  - "address" → street address

OPTIONAL but useful:
  - "description" → project description
  - "estimated_cost" → job value / estimated cost
  - "contact_name" → contractor or applicant name
  - "owner_name" → property owner
  - "latitude" / "longitude" → coordinates
  - "status" → permit status
  - "issue_date" → date issued (if different from filing_date)
  - Any other city-specific fields that appear in the data

DO NOT include a field in field_map if:
  - The field doesn't exist in the source data
  - You're not sure what it maps to
  - It's not useful for permit research

The "date_field" at the TOP LEVEL MUST match the SOURCE field name.
This is what the collector uses for date-range filtering.
Example: if the source uses "IssuedDate", then:
  - "date_field": "IssuedDate"  (top level, source name)
  - "filing_date": "IssuedDate" (in field_map, maps to source)

### 3.5 LAT/LON COORDINATES

Every city config MUST have "lat" and "lon" fields. These should be well-known
coordinates for the city center (used for geolocation on the website).

You can find these by:
  - Google: "[city name], [state]" — look at the coordinates
  - OpenStreetMap
  - Common knowledge (e.g., Chicago's downtown is ~41.88, -87.63)
  - City hall address coordinates

Precision: 3-4 decimal places is fine (e.g., 41.878, -87.630)

### 3.6 SLUG FORMAT

The slug is used in URLs: permitgrab.com/permits/[slug]

Format rules:
  - lowercase
  - hyphens (no underscores)
  - no state suffix UNLESS there's ambiguity
  - max 40 characters

Examples:
  - Los Angeles, CA → "los-angeles"
  - Portland, OR → "portland"
  - Portland, ME → "portland-me"
  - San Jose, CA → "san-jose"
  - New York City, NY → "new-york"

## 4. HOW TO VALIDATE A DATA SOURCE BEFORE ADDING CONFIG
================================================================================

Before adding a config to city_configs.py, Code MUST:

1. Make a test HTTP request to the endpoint
2. Verify the response contains actual permit data (not error)
3. Check that the date field has RECENT data (within last 6 months)
4. Identify the correct field names from the response JSON
5. Verify it's building/construction permits (not business licenses, zoning, etc.)

### 4.1 SOCRATA TEST

```bash
curl -s "https://data.domain.com/resource/xxxx-xxxx.json?\$limit=3&\$order=date_field+DESC" | python3 -m json.tool | head -50
```

You should see:
  - 3-10 JSON objects (permits)
  - Each with field names you can map
  - A date field with RECENT dates (2026-02 or later)
  - No error messages

### 4.2 ARCGIS TEST

```bash
curl -s "https://services.arcgis.com/.../FeatureServer/0/query?where=1%3D1&outFields=*&resultRecordCount=3&orderByFields=DateField+DESC&f=json" | python3 -m json.tool | head -100
```

You should see:
  - "features": [ ... ] array with 3-10 objects
  - Each object has "attributes": { field_name: value, ... }
  - No "error" key at the top level
  - No HTTP error (should be 200)

### 4.3 CKAN TEST

```bash
curl -s "https://data.domain.com/api/3/action/datastore_search?resource_id=UUID&limit=3&sort=date_field+desc" | python3 -m json.tool | head -50
```

You should see:
  - "success": true
  - "result": { "records": [ ... ] }
  - 3-10 records in the array
  - Each with field names matching the dataset schema
  - RECENT dates in the date field

### 4.4 COMMON VALIDATION FAILURES

1. Empty response (no records)
   → The endpoint is working but there's no data matching the date filter.
   → Check if the date field name is correct.
   → Check if the dataset is really building permits (not empty).

2. 403 Forbidden / 401 Unauthorized
   → The dataset requires authentication.
   → Mark as "# LOGIN_REQUIRED: Dataset requires API key or login"

3. 404 Not Found
   → The endpoint is dead. Try searching for a replacement.
   → If no replacement exists, mark as "# BLOCKED: Endpoint removed/404"

4. JSON parse error or HTML response
   → The endpoint doesn't return JSON.
   → It might be a web form or PDF.
   → Try a different endpoint or mark as BLOCKED.

5. Recent dates are OLD (oldest is 2024-01 or earlier)
   → The dataset may be discontinued.
   → Check if the city has a newer source.
   → If not, mark as "# BLOCKED: Dataset last updated [date], no active feed"

6. Rows return but all nulls in key fields
   → The field names don't match what the collector expects.
   → Inspect the actual field names from the response.
   → Update the field_map accordingly.

### 5.1 STALE CITIES (PRIORITY 1)

These cities ALREADY have a config in city_configs.py but data is old
(last_data is 30+ days in the past).

Likely issues:
  - Endpoint URL changed or dataset was migrated
  - Started requiring authentication
  - Dataset was taken offline
  - Date field name changed

Process for each STALE city:

1. Open city_configs.py and find the existing config for this city
   - Note the current endpoint, date_field, and field_map

2. Test the current endpoint:
   - Run the curl command from section 4
   - Does it return data? Any errors?

3. If the endpoint WORKS:
   - Check if the date field still has recent data
   - If yes, the issue might be:
     a) The date_field name changed (different field now has recent data)
     b) The site.endpoint still works but data is somehow stale
   - Update the config if needed and mark as "V23 AUDIT: 2026-03-28 - FIXED"

4. If the endpoint is DEAD (404, 403, or doesn't return data):
   - Search for a replacement dataset on the same city domain
   - Check if the city migrated to a new portal
   - Google: "[city name]" open data building permit (find new URL)
   - If found, update the endpoint URL and test it

5. If NO alternative endpoint exists:
   - Mark with: "# BLOCKED: Original endpoint offline, no public replacement found"
   - Add V23 AUDIT comment

### 5.2 NO_DATA CITIES (PRIORITY 2)

These cities HAVE a config and are marked active: True, but have collected
0 permits since creation. The config is likely WRONG.

Likely issues:
  - Wrong endpoint URL entirely
  - Wrong field_map (field names don't match the actual data)
  - Wrong date_field
  - Endpoint returns data but NOT building permits (it's business licenses, etc.)
  - Data is present but all dates are in the past (old dataset)

Process for each NO_DATA city:

1. Open city_configs.py and find the existing config

2. Test the endpoint directly:
   - Run the curl command from section 4
   - What does it actually return?

3. If it returns ERROR (403, 404, etc.):
   - Same as STALE: search for replacement
   - Or mark as BLOCKED if none exists

4. If it returns DATA but looks WRONG:
   - Check if the field names in field_map match the actual response
   - If "permit_number" field is "PERMITS_ID" in response, that's a mismatch
   - Fix the field_map
   - Verify the data_field is correct

5. If it returns DATA but ALL DATES ARE OLD:
   - The dataset exists but is abandoned (last update 2024-11, no new permits)
   - Mark as "# BLOCKED: Dataset exists but last update [date], no active permits"

6. If it returns DATA and looks CORRECT:
   - The issue might be with how the collector processes it
   - Run a manual test: python3 -c "from collector import fetch_permits; data, status = fetch_permits('city_key', days_back=180); print(f'{len(data)} permits, status: {status}')"
   - Check the status message
   - If it works now, mark as "V23 AUDIT: 2026-03-28 - REACTIVATED"

7. Save, commit with message: "V23: Batch N - Fixed [X] NO_DATA cities"


### 5.3 INACTIVE CITIES (PRIORITY 3)

Process for each INACTIVE city:

   - If no notes, it's a mystery — assume endpoint was broken

2. Test the endpoint:

3. If it WORKS and returns RECENT data:

   - Add notes: "V23 AUDIT: 2026-03-28 - REACTIVATED - Endpoint working"

4. If it STILL DOESN'T WORK:
   - Try to find a replacement source
   - If found, update endpoint and set active: True
   - If not found, add "# BLOCKED:" comment explaining why

5. If it WORKS but ALL DATA IS OLD:
   - Mark as "# BLOCKED: Endpoint works but data last updated [date]"

6. Save, commit with message: "V23: Batch N - Processed [Y] INACTIVE cities"

## 6. FIELD MAP GUIDELINES (DETAILED)
================================================================================

The field_map is the most critical part of the config. It tells the collector
how to extract permit data from the source API response.

### REQUIRED FIELDS

"filing_date" (CRITICAL):
  - Maps to the source's date field
  - Used for filtering (only get permits from last 180 days)
  - Must always be present
  - Example: "filing_date": "issued_date"

### IMPORTANT FIELDS (Include if available)

"permit_number":
  - The unique permit ID
  - Examples: "PERMIT_ID", "PermitNum", "permitnumber", "job__"
  - If not available, use blank string: "permit_number": ""

"permit_type":
  - Type of permit (residential, commercial, demolition, etc.)
  - Examples: "PERMIT_TYPE", "permittypemapped", "work_type"
  - If not available, use blank string

"address":
  - Street address of the property
  - Examples: "street_address", "SITE ADDRESS", "primary_address"
  - If not available, try combining "street_number" + "street_name"

### OPTIONAL BUT USEFUL FIELDS

"description":
  - Project description / scope of work
  - Examples: "work_description", "PROJECT DESCRIPTION", "job_description"

"estimated_cost":
  - Job value or project valuation
  - Examples: "project_value", "VALUATION", "estimated_job_costs"

"contact_name":
  - Contractor or applicant name
  - Examples: "contractorname", "APPLICANT_NAME", "applicant_business_name"

"owner_name":
  - Property owner name
  - Examples: "OWNER_NAME", "owner_business_name"

"latitude" / "longitude":
  - Coordinates (if available)
  - Examples: "lat", "LATITUDE", "gis_latitude"

"status":
  - Permit status (approved, issued, pending, etc.)
  - Examples: "permit_status", "APPLICATION_STATUS_NAME", "statuscurrent"

"issue_date":
  - Date permit was issued (if different from filing_date)
  - Examples: "issued_date", "issue_date", "approval_date"

### HOW TO FIND FIELD NAMES

1. Run the test curl command and look at the actual response JSON
2. Example Socrata response:
   ```json
   {
     "permit_id": "2024-123456",
     "permit_type": "Building Permit",
     "address": "123 Main St",
     "issued_date": "2026-03-20T00:00:00",
     "work_description": "New construction, 5-story office"
   }
   ```
   Map these to standard fields:
   - "permit_number": "permit_id"
   - "permit_type": "permit_type"
   - "address": "address"
   - "filing_date": "issued_date"
   - "description": "work_description"

3. Be EXACT with field name capitalization:
   - "IssuedDate" is different from "issued_date"
   - "CITY_NAME" is different from "city_name"
   - Check the actual response

4. When in doubt, test the collector:
   python3 -c "from collector import fetch_permits; data, status = fetch_permits('city_key', days_back=1); print(data[0].keys() if data else 'no data')"

### TOP-LEVEL DATE_FIELD

The "date_field" at the config top level MUST be the SOURCE field name.
This is what the collector uses in the WHERE clause for date filtering.

```python
"config_key": {
    ...
    "date_field": "issued_date",        # SOURCE field name (from API)
    "field_map": {
        "filing_date": "issued_date",   # Standard field maps to source field
        ...
    }
}
```

These must MATCH. If they don't, the collector won't filter by date.

## 7. COORDINATES (LAT/LON)
================================================================================

Every city config MUST have "lat" and "lon" fields (in decimal degrees).

These are used for:
  - Geolocation on the PermitGrab website
  - Geographic clustering in the admin dashboard
  - Future location-based features

How to find coordinates:

1. Google: "[city name], [state]"
   - Google Maps shows coordinates at the bottom
   - Copy the lat/lon

2. OpenStreetMap: openstreetmap.org
   - Search for the city
   - Right-click on the map center
   - Copy coordinates

3. City Hall address
   - Find the address of city hall
   - Google the coordinates
   - Use that as the city center

4. Common knowledge:
   - New York City: 40.713, -74.006
   - Los Angeles: 34.052, -118.244
   - Chicago: 41.878, -87.630
   - Houston: 29.761, -95.369
   - Phoenix: 33.448, -112.074

Precision: 3-4 decimal places is fine. Don't over-specify.

Example:
```python
"lat": 40.713,   # New York latitude
"lon": -74.006,  # New York longitude
```

## 8. SLUG FORMAT
================================================================================

The slug is used in URLs: permitgrab.com/permits/[slug]

Slug rules:
  - Lowercase only
  - Hyphens for word separators (NO underscores, NO spaces)
  - Max 40 characters
  - No ambiguity problems
  - Use state suffix ONLY if two cities have the same name

Examples:
  - "Los Angeles, CA" → "los-angeles"
  - "Portland, OR" → "portland"
  - "Portland, ME" → "portland-me"
  - "Fort Worth, TX" → "fort-worth"
  - "New York City, NY" → "new-york"
  - "San Jose, CA" → "san-jose"
  - "St. Paul, MN" → "st-paul"

If you're unsure, check existing city_configs.py for the pattern.
Most cities DON'T have state suffix (only when there's ambiguity).

## 11. BATCHING STRATEGY
================================================================================

Process cities in batches of approximately 15-25 at a time.

Why batches?
  - Reduce commit risk (smaller, focused changes)
  - Allow easy rollback if a batch has issues
  - Clear progress tracking
  - Easier to test individual cities

Batch process:

1. Load worklist.txt
2. Sort by priority tier + population (descending)
3. Pick next 20 cities from the sorted list
4. For each city:
   - Discover/validate data source (section 2 & 4)
   - Create or fix config (section 3)
   - Add V23 AUDIT comment
   - Set "active": True or add "# BLOCKED:" note

5. Open city_configs.py and apply all changes for the batch
6. Spot-check 3-5 configs by running the test command
7. Save and commit:
   ```
   git add city_configs.py
   git commit -m "V23: Audit batch [N] - [summary]"
   ```

8. Move to next batch

Expected timeline:
  - ~5-10 cities per hour (includes research, validation, config writing)
  - 1,898 cities / 7 cities per hour = ~270 hours of work
  - With concurrent processing, aim to complete all tiers in ~2-3 weeks

## 12. IMPORTANT NOTES & SPECIAL CASES
================================================================================

### A. CALIFORNIA CITIES
Many California cities share data through statewide portals:
  - State of California Open Data (data.ca.gov)
  - Individual county portals (data.county.ca.us)
  - Regional council of governments portals

Often need "city_filter" in config to narrow down to specific city.

### B. NEW JERSEY CITIES
Most NJ cities use NJ's statewide Socrata portal (nj.gov/opendata or similar).
Use config option to filter by city name:

```python
"city_filter": {
    "field": "CITY_NAME",
    "value": "Newark"
}
```

### C. TEXAS CITIES
Many Texas cities are in regional councils of governments (COGs):
  - North Central Texas Council of Governments
  - NCTCOG Building Permits portal
  - May share the same dataset with city-level filtering

Check city's website for the primary data source.

### D. EPOCH TIMESTAMPS
ArcGIS often uses epoch milliseconds instead of date strings.

Epoch milliseconds = Unix timestamp * 1000
Example: 1711929600000 = March 31, 2024 16:00:00 UTC

In config, set: "date_format": "epoch"

### E. EMPTY DATASETS
Some cities create open data portals but never populate them.
After 2-3 test calls with no data, mark as:
```
"# BLOCKED: Public dataset exists but is empty (0 records)"
```

### F. CITY_FILTER SYNTAX
Some platforms (especially CKAN) require exact syntax for city filters.

Socrata:
```python
"city_filter": {
    "field": "CITY_NAME",
    "value": "Newark"
}
# Becomes: AND upper(CITY_NAME) = upper('Newark')
```

CKAN:
```python
"city_filter": {
    "field": "CITY_NAME",
    "value": "Newark"
}
# Becomes: Python string comparison (case-insensitive)
```

### G. ARCHIVED/DISCONTINUED DATASETS
Some cities take datasets offline when they upgrade systems.
If you find an old Socrata dataset that returns 404:
  - Check if the city migrated to a new platform
  - Check if they moved the dataset to a different domain
  - Mark as BLOCKED only after checking the city's current website

### H. LOGIN-REQUIRED ENDPOINTS
If the only source requires API key registration:

```python
"notes": "# LOGIN_REQUIRED: Portal requires API key registration at https://[city].gov/opendata/register"
```

These will be processed manually later through Chrome.

### I. TESTING DATE RANGES
Always test with a date range that should return data:

```bash
# Test last 6 months
curl -s "https://data.domain.com/resource/xxxx-xxxx.json?\$limit=3&\$where=issued_date%20%3E%20%272025-09-28%27&\$order=issued_date+DESC" | python3 -m json.tool
```

If this returns 0 records, try:
  - Last year: 2025-03-28
  - Last 2 years: 2024-03-28
  - All records: remove $where clause

This tells you if the dataset is actively maintained.

### J. INCONSISTENT FIELD NAMES
Some APIs use different field names in different responses or across pages.
Always validate with the ACTUAL response, not documentation.

### K. PERFORMANCE & RATE LIMITS
Most open data APIs don't have aggressive rate limits, but some do:
  - Don't hammer the API during discovery
  - Use limit=3-5 for test requests
  - If you hit rate limits, wait 1-2 minutes before retrying

### L. FIELD NAME CASE SENSITIVITY
APIs differ in case sensitivity:
  - Socrata: lowercase fields (e.g., "permit_id")
  - ArcGIS: often PascalCase (e.g., "PermitID")
  - CKAN: often UPPERCASE (e.g., "PERMIT_ID")

Always check the ACTUAL response to get the exact case.

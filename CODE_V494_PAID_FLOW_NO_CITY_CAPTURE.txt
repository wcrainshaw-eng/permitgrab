ROOT CAUSE
==============================================================================

The current flow for a paid subscription:

    /pricing
       └─ "Start Free Trial" button
              ↓
    /start-checkout?plan=pro          (routes/city_pages.py:723)
       ├─ if logged-out:
       │     redirect /signup → /start-checkout
       └─ if logged-in:
              ↓
    Stripe Checkout                    (creates subscription)
              ↓
    Stripe webhook (checkout.session.completed)
              ↓
    User row updated: plan=pro
              ↓
    [NO subscribers row written. No digest_cities captured.]

The `/get-alerts` route (city_pages.py:805) and the `/api/subscribe`
endpoint (api.py:871) capture city preferences — but ONLY for free
digest signups. Paid users never go through them.

The "patch" Wes mentioned (probably an earlier attempt to wire
/get-alerts before checkout) is NOT in the live code. /start-checkout
sends paid users straight to Stripe with no city capture step.

This means:
  - Every new paid customer enters with digest_cities=NULL
  - The digest scheduler skips users with no digest_cities
  - The scheduler subscriber list shows 2 users today, regardless
    of how many people actually paid

THE STRUCTURAL FIX
==============================================================================

The flow has to capture city prefs BEFORE handing the user to Stripe.
There are two viable patterns:

Pattern A — pre-checkout form (recommended)
    /pricing
       └─ "Start Free Trial"
              ↓
    /signup (if logged-out) → /start-checkout
              ↓
    /select-cities?next=/start-checkout?plan=pro    (NEW route)
       └─ form POST to /select-cities (NEW POST handler)
              ↓ (form writes pending subscribers row + redirects)
    /start-checkout (now has session.cities populated)
              ↓
    Stripe Checkout
              ↓
    webhook → activate subscribers row from session.cities

Pattern B — Stripe Custom Field
    Stripe Checkout supports custom fields. Add a "Which cities do
    you want alerts for?" multi-select field to the Checkout Session
    creation. The webhook then reads custom_fields off the session
    and writes the subscribers row.
    Cleaner UX (one less hop) but Stripe's custom field UI is plain
    text only — no city-name validation, no autocomplete. Risky.

Pattern A is the right call. Implementation:

==== EDIT 1: routes/city_pages.py — add /select-cities route ====

Add after /get-alerts (~line 820):

    @city_pages_bp.route('/select-cities')
    def select_cities_page():
        """V494: city-prefs capture step that gates Stripe checkout.

        Required step BEFORE /start-checkout for any paid plan.
        Without this, paying customers land in Stripe with no
        digest_cities captured, the webhook can't write a subscribers
        row, and they're charged but receive no digest.

        URL params:
          plan: pro/starter/enterprise (default pro)
          next: where to redirect after submission (default /start-checkout)
        """
        from flask import session
        user = get_current_user()
        if not user:
            # Shouldn't reach here — /pricing button gates on /signup —
            # but defensive in case someone deep-links.
            from urllib.parse import quote
            next_url = quote(request.full_path, safe='')
            return redirect(f'/signup?next={next_url}')

        plan = (request.args.get('plan') or 'pro').strip().lower()
        next_path = request.args.get('next') or f'/start-checkout?plan={plan}'

        # If they already have cities captured (returning user
        # upgrading from free → paid), skip straight to checkout.
        existing = permitdb.execute(
            "SELECT digest_cities FROM subscribers WHERE email = ?",
            (user.get('email'),)
        ).fetchone()
        if existing and existing[0] and existing[0] != '[]':
            return redirect(next_path)

        all_cities = get_cities_with_data()
        cities = all_cities[:200]
        return render_template('select_cities.html',
                               cities=cities,
                               plan=plan,
                               next_path=next_path,
                               user=user)


    @city_pages_bp.route('/select-cities', methods=['POST'])
    def select_cities_submit():
        """V494: write a pending subscribers row + redirect to checkout."""
        from flask import session
        user = get_current_user()
        if not user:
            return redirect('/signup')

        cities = request.form.getlist('cities')
        if not cities:
            return redirect('/select-cities?error=at_least_one_city_required')

        plan = (request.form.get('plan') or 'pro').strip().lower()
        next_path = request.form.get('next_path') or f'/start-checkout?plan={plan}'

        # Stash in session AND write a pending subscribers row. The
        # session is the primary path; the DB row is a backup so we
        # can recover if the user closes the browser before completing
        # Stripe checkout.
        import json as _j
        email = user.get('email')
        name = user.get('name') or ''
        permitdb.execute(
            "INSERT OR REPLACE INTO subscribers "
            "(email, name, plan, digest_cities, active, created_at) "
            "VALUES (?, ?, ?, ?, 0, datetime('now'))",
            (email, name, plan, _j.dumps(cities))
        )
        session['pending_cities'] = cities
        session['pending_plan'] = plan

        return redirect(next_path)

Find the webhook handler at routes/api.py:2620 (the

    # V494: activate the pending subscribers row written by
    # /select-cities, AND defensively sync plan on any existing
    # row matching this email. The current `subscribers` table only
    # contains hand-inserted rows because no real user has ever
    # completed an organic signup-to-digest flow before; the webhook
    # has never written to subscribers. After this patch, every
    # checkout.session.completed event keeps Postgres (users.plan)
    # and SQLite (subscribers.plan) in sync.
    customer_email = (session_obj.get('customer_email') or
                      session_obj.get('customer_details', {}).get('email') or
                      '').strip().lower()
    if customer_email:
        # Sync plan on any existing subscribers row
        # (defensive — covers free-to-paid upgrades + manual back-fills)
        permitdb.execute(
            "UPDATE subscribers SET plan = ?, "
            "  last_updated = datetime('now') "
            "WHERE LOWER(email) = ?",
            (plan_from_metadata, customer_email)
        )
        # Activate any pending row from /select-cities
        # (the primary case — net-new paid signups via the V494 flow)
        permitdb.execute(
            "UPDATE subscribers SET active = 1 "
            "WHERE LOWER(email) = ? AND active = 0",
            (customer_email,)
        )
        # If no pending row existed (legacy path or session lost),
        # create one with NO cities — this signals the customer-success
        # flow that we need to email them for their preferences.
        existing = permitdb.execute(
            "SELECT id FROM subscribers WHERE LOWER(email) = ?",
            (customer_email,)
        ).fetchone()
        if not existing:
            permitdb.execute(
                "INSERT INTO subscribers "
                "(email, name, plan, digest_cities, active, created_at) "
                "VALUES (?, ?, ?, '[]', 1, datetime('now'))",
                (customer_email, '', plan_from_metadata)

WHY THIS HAPPENED
==============================================================================

V12.51 (a long time ago) split signup into auth (User row) and
digest (subscribers row) but never wired the paid path through the
digest form. The /get-alerts route exists for free digest signups
and /pricing exists for paid signups — they were two separate funnels
that never converged. Free digest users get city prefs because their
ONLY entry is /get-alerts. Paid users skip /get-alerts entirely.

V375 (CODE_V363) "unified" the Start Free Trial button across
logged-in and logged-out states — but didn't fix the missing
city-capture step.

This bug has probably been silent on every paid signup since the
free→paid divergence. Wes + Benjamin showing up correctly on the
digest is because Wes manually configured both rows (Wes is admin
+ founder; Benjamin is a friend who signed up on the free side
first then... probably never upgraded to pro? Worth verifying).

In short: the system has been billing customers without delivering
the product since the first paid customer joined. We just didn't
notice because Wes was the only paid customer until very recently.

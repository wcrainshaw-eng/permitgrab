(Pre-Construction Intelligence / Early Signals)

Read ~/Documents/PermitGrab/CLAUDE_CODE_REFERENCE.txt for full context.

This batch adds a NEW data type called "signals" — pre-construction intelligence that arrives BEFORE permits are filed. These are zoning applications, planning commission decisions, variance requests, and early-stage building filings. They don't have permit numbers, so they live in a separate data model connected to permits only by address matching.

IMPORTANT ARCHITECTURE NOTE: Signals are NOT permits. They are a parallel data layer stored in data/signals.json. Each signal has its own ID format (signal_CITY_TIMESTAMP_HASH). The connection between signals and permits happens through normalized address matching — when a permit eventually appears at the same address as a signal, we link them.

Add these features to PermitGrab:

2. SIGNAL COLLECTOR (signal_collector.py):
   Create a new file signal_collector.py that pulls pre-construction signals from free city open-data APIs:

   NYC (4 sources):
   a) ZAP — Zoning Application Portal
      - API: https://data.cityofnewyork.us/resource/2iga-a6mk.json
      - Pull: project_name, project_brief, borough, primary_address, applicant_name, project_status
      - Filter: $where=certified_referred > '(90 days ago)' OR project_status='Active'
      - Signal type: "zoning_application" or "land_use_review"

   b) BSA Applications — Board of Standards & Appeals
      - API: https://data.cityofnewyork.us/resource/yvxd-uipr.json
      - Pull: application_number, premises, block, lot, calendar_status
      - Filter: Recent 90 days
      - Signal type: "variance_request"

   c) DOB Job Application Filings (early-stage, before permit issuance)
      - API: https://data.cityofnewyork.us/resource/ic3t-wcy2.json
      - Pull: job_number, house_number, street_name, borough, job_type, job_status, owner_name, estimated_job_cost
      - Filter: $where=pre_filing_date > '(60 days ago)' AND job_status IN ('A','D','E','G','H','J','K')
      - Signal type: "new_building_filing" (if job_type='NB') or "demolition_filing" (if job_type='DM')
      - NOTE: Only pull filings that do NOT yet have an approved permit (job_status not in 'R' which means issued)

   d) DOB New Buildings
      - API: https://data.cityofnewyork.us/resource/6xbh-bxki.json
      - Pull: job_number, address, borough, owner_name, estimated_cost
      - Filter: Recent 90 days, status != 'permit issued'
      - Signal type: "new_building_filing"

   CHICAGO (2 sources):
   a) Affordable Housing Pipeline
      - API: https://data.cityofchicago.org/resource/s6ha-ppgi.json
      - Pull: address, community_area, units, property_name, developer
      - Signal type: "planning_approval"

   b) Building Permits (pre-issue status)
      - API: https://data.cityofchicago.org/resource/ydr8-5enu.json
      - Filter: $where=permit_status='ISSUE' AND application_start_date > '(60 days ago)' AND permit_type='PERMIT - NEW CONSTRUCTION'
      - Signal type: "new_building_filing"

   LOS ANGELES:
   a) Building & Safety Permit Application
      - API: https://data.lacity.org/resource/yv23-pmwf.json
      - Filter: applications in "Plan Check" status (not yet permitted)
      - Signal type: "new_building_filing"

   AUSTIN:
   a) Issued Construction Permits (use as early signal for large projects)
      - API: https://data.austintexas.gov/resource/3syk-w9eu.json
      - Filter: permit_type='BP' (Building Permit) with status_current='In Review'
      - Signal type: "new_building_filing"

   DENVER:
   a) Denver Development Pipeline
      - API: https://data.denvergov.org has development applications
      - Pull development review applications
      - Signal type: "planning_approval"

   SEATTLE:
   a) Building Permits - In Review
      - API: https://data.seattle.gov/resource/76t5-zuj7.json
      - Filter: $where=status_current='In Review' or status_current='Application Accepted'
      - Signal type: "new_building_filing"

   PORTLAND:
   a) Building Permits - Under Review
      - API: https://data.portland.gov - BDS permits dataset
      - Filter: status = 'Under Review' or 'Submitted'
      - Signal type: "new_building_filing"

   SAN FRANCISCO:
   a) DBI Building Permit Applications
      - API: https://data.sfgov.org/resource/i98e-djp9.json
      - Filter: status='filed' or status='plancheck' (before issuance)
      - Signal type: "new_building_filing"

   For EACH source, the collector should:
   - Normalize addresses to lowercase, strip extra spaces, append city+state
   - Deduplicate against existing signals by address + signal_type + source_dataset
   - Check if any existing PERMITS in data/permits.json match the signal's address — if so, add the permit_id to linked_permits[]
   - Log collection stats to data/signal_collection_stats.json

3. ADDRESS MATCHING ENGINE:
   - Create a function match_signals_to_permits() in signal_collector.py that:
     * Normalizes both permit addresses and signal addresses the same way (lowercase, strip apt/unit/suite, strip punctuation, expand abbreviations like St→Street)
     * On each collection run, scans all unlinked signals and checks for matching permits
     * When a match is found, updates signal.linked_permits[] with the permit_id
     * Also updates the permit data to include a "linked_signals" field pointing back
   - This runs after both the permit collector and signal collector finish

4. API ENDPOINTS:
   - GET /api/signals?city=X — returns all signals for a city, sorted by date_filed desc
   - GET /api/signals?city=X&type=zoning_application — filter by signal type
   - GET /api/signals/<signal_id> — returns a single signal with linked permits
   - GET /api/signals/stats?city=X — returns signal counts by type and status
   - GET /api/address-intel/<address> — returns ALL intelligence for an address: permits, signals, violations (from Batch 2), and permit history (from Batch 2). This is the "full picture" endpoint.

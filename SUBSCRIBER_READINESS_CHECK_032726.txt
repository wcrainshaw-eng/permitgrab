BUGS TO FIX (RED — subscriber-visible)
================================================================================

BUG #1: DUPLICATE PERMITS ON HOMEPAGE (HIGH PRIORITY)
  What: "22 WELCOME PL SW, Michigan, MI" appears TWICE in the main feed.
        Once at $11.6M, once at $26.9M. Same address, same date, same trade.
  Impact: First thing a subscriber sees. Looks sloppy.
  Root cause: 3,453 excess duplicate rows in permits table (7.1% of all data).
        Same address + city + filing_date appearing multiple times.
        Worst offender: "125 KINSALE, Oxon Hill, MD" appears 62 times.
  Fix: Add dedup logic on INSERT — unique constraint on
       (address, city, state, filing_date, permit_type) or dedup on display.
       Also run a one-time cleanup to DELETE excess duplicates from permits table.

BUG #2: GARBAGE CITY NAMES (HIGH PRIORITY)
  What: 2,797 permits (5.7%) have numeric city names: "6" (AZ), "5.0" (VA),
        "2.0" (VA), "4.0" (VA), etc.
  Impact: These show up in the main feed. "6, AZ" is not a city.
  Root cause: AZ source has district numbers in the city field.
        VA source has some numeric rating/code in the city field.
  Fix: Add city name validation — reject any city that's purely numeric
       or a decimal. Clean up existing records by mapping to actual cities
       (the AZ addresses are Mesa/Gilbert/Chandler area, VA addresses are
       Virginia Beach/Norfolk area based on the addresses).

BUG #3: NEIGHBORHOODS AS CITIES (MEDIUM PRIORITY)
  What: 3,673 permits (7.5%) list Orlando FL neighborhoods as cities:
        "Lake Nona South", "Vista Park", "College Park", "Meridian Park",
        "Florida Center", "Southeastern Oaks", "Central Business District",
        "Johnson Village", "33Rd St. Industrial"
  Impact: "Central Business District, FL" looks wrong to anyone.
  Root cause: Orlando/Orange County bulk source uses neighborhoods in city field.
  Fix: Map these neighborhood names to "Orlando" in the collector.
       Create a NEIGHBORHOOD_TO_CITY lookup table.

BUG #4: "CALGARY" IN LEADS BY CITY SIDEBAR (MEDIUM PRIORITY)
  What: "Calgary" appears with 135 leads in the "Leads by City" sidebar.
  Impact: Canadian city shouldn't be in a US permit leads product.
  Root cause: Likely from a bulk source that includes Canadian data,
        or a city name parsing error.
  Fix: Filter out non-US cities. Add country validation.

BUG #5: /api/health RETURNS 404 (LOW PRIORITY)
  What: https://permitgrab.com/api/health returns "Page Not Found"
  Impact: Not subscriber-visible, but useful for monitoring.
  Root cause: Route was added in V18 code but may not be deployed yet.
  Fix: Verify the route is in the deployed version on Render.

BUG #6: "ADDRESS NOT PROVIDED" — 22% OF PERMITS (MEDIUM PRIORITY)
  What: 10,781 permits (22.1%) have "Address not provided" or empty address.
  Impact: Useless to contractors — they need to know WHERE the job is.
  Root cause: NJ bulk source doesn't include addresses for many permits.
  Fix: Either hide these from the default view, or add a filter to exclude
       "no address" permits. At minimum, don't show them in "Best Leads" sort.

BUG #7: ZERO-PERMIT ACTIVE CITIES (MEDIUM PRIORITY)
  What: NYC, SF, Seattle, Nashville, Mesa are status='active' in prod_cities
        but show total_permits=0. These city pages exist but have no data.
  Impact: Subscriber clicks "New York City" and sees nothing.
  Fix: Either populate with actual permit data (sources exist — see
       TOP50_MISSING_SOURCES.txt) or set status='pending' until data loads.

BUG #8: FORT WORTH ADDRESS QUALITY (LOW PRIORITY)
  What: Some Fort Worth permits show just a street name without number:
        "MADRONE" (15 permits), "PORT ARTHUR" (12 permits)
  Impact: Incomplete addresses — not useful for finding the job site.
  Root cause: Fort Worth Accela source sometimes returns street names
        without house numbers for new subdivision permits.
  Fix: Append "NEW SUBDIVISION" or similar label to these incomplete addresses.

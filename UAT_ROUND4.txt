ISSUE 2 — HIGH: Oklahoma still has 1,112 cities (all Texas)
-------------------------------------------------------------
The registry lookup fixed ~44 cities that had CITY_REGISTRY entries
(Houston, Dallas, Austin, etc.), but 1,112 "OK" cities remain.
These are real Texas cities with NO config: Katy, Spring, El Paso,
Corpus Christi, Brownsville, Laredo, Cypress, Midland, etc.

The root cause: some past bulk collection run wrote "OK" as the
state for ~1,156 Texas cities. Only ~44 of those have CITY_REGISTRY
entries to override.

FIX OPTIONS (pick one):

  Option A — DB migration (BEST, one-time fix):
    Run a SQL UPDATE on the Render Postgres DB to correct the state:
      UPDATE permits SET state = 'TX' WHERE state = 'OK'
        AND city NOT IN ('Oklahoma City', 'Tulsa', 'Norman',
            'Broken Arrow', 'Edmond', 'Lawton', 'Moore',
            'Midwest City', 'Enid', 'Stillwater');
    This is aggressive but OK isn't actually used for real Oklahoma
    cities (only Oklahoma City has an active config).

    SAFER version — only fix cities that are KNOWN Texas cities:
      UPDATE permits SET state = 'TX' WHERE state = 'OK'
        AND city IN (select the top 50 TX cities by name);

  Option B — Heuristic in get_cities_with_data():
    If a city appears under "OK" with > 100 permits and does NOT
    match any known Oklahoma city name, reassign it to "TX".
    Known OK cities: Oklahoma City, Tulsa, Norman, Broken Arrow,
    Edmond, Lawton, Moore, Midwest City, Enid, Stillwater.
    Everything else under "OK" is almost certainly TX.

  Option C — Accept it for now:
    The 1,112 "OK" entries are real cities with real permits.
    They just have the wrong state. This is a data quality issue
    that doesn't break functionality — just makes the state
    directory inaccurate. Can be fixed later with a proper
    data cleanup.

  RECOMMENDATION: Option A (DB migration). It's a one-time fix
  that solves the problem at the source. The data in the DB is
  simply wrong — "OK" should be "TX" for these rows.

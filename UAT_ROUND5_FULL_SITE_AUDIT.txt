BUG C2 — State corruption beyond OK: NV has 98 Texas towns
-----------------------------------------------------------
Page: https://permitgrab.com/cities (NV section)
What: The NV (Nevada) section has 100 cities, but ~98 are Texas towns:
      Fredericksbrg (misspelled Fredericksburg TX), Alleyton, Ranger,
      Windthorst, Mountain Home, Colorado City, College Sta (College
      Station TX), Bigfoot, Turkey, Quitaque, etc.
      Only Henderson and Reno are actual Nevada cities.
Impact: Nevada state directory is completely wrong. Same root cause as
        the OK→TX issue but unfixed.
Fix: Apply same KNOWN_OK_CITIES pattern for NV:
     KNOWN_NV_CITIES = {"Las Vegas", "Henderson", "Reno", "North Las Vegas",
       "Sparks", "Carson City", "Elko", "Mesquite", "Boulder City", "Fernley"}
     If state='NV' and city not in KNOWN_NV_CITIES → reassign to 'TX'
     OR: Fix at DB level with UPDATE permits SET state='TX' WHERE state='NV'
         AND city NOT IN ('Las Vegas','Henderson','Reno',...);


BUG C3 — State corruption: IN has 25 Florida cities, LA has 70 CA cities
-------------------------------------------------------------------------
Page: https://permitgrab.com/cities (IN and LA sections)

IN (Indiana) — 28 cities, ~25 are Florida:
  Miami (677), North Miami Beach, Doral, Boca Raton, Cutler Bay,
  Miami Lakes, Coral Gables, Miami Shores, Homestead, Tamarac, Apopka
  Also "Uninc" (48) — not a city name ("unincorporated")
  Only Bloomington could plausibly be Indiana.

LA (Louisiana) — 72 cities, ~70 are Los Angeles CA neighborhoods:
  Tarzana, Northridge, Sherman Oaks, Encino, Woodland Hills, Simi Valley,
  Sylmar, Studio City, Valencia, West Hills, North Hollywood, Ontario,
  Panorama City, etc.
  Only New Orleans and Baton Rouge are actual Louisiana cities.
  Root cause: "LA" state code confused with "LA" as Los Angeles.

Impact: State directories for IN and LA are completely wrong.
Fix: Same heuristic pattern as KNOWN_OK_CITIES:
     KNOWN_IN_CITIES = {"Indianapolis", "Fort Wayne", "Evansville",
       "South Bend", "Carmel", "Fishers", "Bloomington", "Hammond",
       "Gary", "Lafayette", "Muncie", "Terre Haute"}
     If state='IN' and city not in KNOWN_IN_CITIES → reassign to 'FL'

     KNOWN_LA_CITIES = {"New Orleans", "Baton Rouge", "Shreveport",
       "Lafayette", "Lake Charles", "Kenner", "Bossier City", "Monroe",
       "Alexandria", "Houma", "Slidell", "Metairie"}
     If state='LA' and city not in KNOWN_LA_CITIES → reassign to 'CA'

     OR: DB-level fix (better long-term).

BUG H1 — Homepage "Leads by City" sidebar shows garbage entries
---------------------------------------------------------------
Page: https://permitgrab.com/ (right sidebar)
What: The sidebar shows non-city entries mixed with real cities:
      - "DOB NOW: Build – Approved" (539 leads) — NYC dataset name
      - "Michigan" (607 leads) — state name, not city
      - "EPIC-LA Fire Recovery Cases" (41)
      - "Building Permit Applications" (38)
      - "EBR" (28)
      - "Maine" (4) — state name
      - "Land Use" (4)
      - "BP_Comm_Dev_Report_Data view" (4)
      - "Commercial" (4), "DSC" (4)
      - "Issued" (17), "Residential Permit" (17), "Permitting Table" (16)
Impact: Makes the homepage look unprofessional and untrustworthy.
Root cause: The "Leads by City" sidebar uses a DIFFERENT code path than
            get_cities_with_data(). It does NOT have the VALID_STATES
            filter or garbage pattern matching.
Fix: Apply the same VALID_STATES filter and garbage-name filter to
     whatever function feeds the homepage sidebar. Or better: make the
     sidebar use get_cities_with_data() directly.


BUG H2 — City count inconsistency across entire site
-----------------------------------------------------
The same "cities covered" number is different on every page:
  Homepage yellow bar:    "4290+ cities covered"
  Pricing "By Numbers":   "4290+"
  Pricing FAQ text:       "4298+ metro areas"
  About page:             "85+ metro areas"
  /cities page title:     "Browse 1601+ Cities"
  TX state hub:           "26 cities" (for Texas alone)

These wildly different numbers destroy credibility. A visitor checking
two pages will immediately notice the discrepancy.

Fix: Pick ONE authoritative number and use it everywhere:
     - If counting DB rows: ~1,601 (from get_cities_with_data)
     - If counting configured cities: ~311 active configs
     - If counting metro areas: ~85 (About page number)
     - The "4290" number appears fabricated or from a stale source
     Make all pages pull from the same source of truth.


BUG H3 — TX has 1,170 cities (massively inflated)
--------------------------------------------------
Page: https://permitgrab.com/cities (TX section)
What: Texas shows 1,170 cities. These are all the formerly-OK cities
      reassigned by the KNOWN_OK_CITIES heuristic. While the state
      assignment is now correct (they ARE Texas cities), having 1,170
      entries makes the page unusable and the count misleading.
      Many are tiny towns with 1-6 permits: Bigfoot (6), Turkey (6),
      Quitaque (5), Runge (5), Anton (4), Bagwell (4), etc.
Impact: TX section is 73% of all cities. The page is dominated by it.
        The "1601+ cities" headline is inflated by ~1,100 tiny TX towns.
Fix: Add a minimum permit threshold (e.g., >= 10 or >= 25 permits)
     to hide tiny entries. Or add pagination/truncation per state
     (show top 20, "View all 1,170 →" link).


BUG H4 — Houston missing from Texas state hub
----------------------------------------------
Page: https://permitgrab.com/permits/texas
What: The Texas state hub shows "26 Cities Covered" and lists them.
      Houston (the largest TX city with 9,043 permits) is NOT listed.
      The hub only shows cities from active CITY_REGISTRY configs.
      Houston's config is INACTIVE, so it's excluded.
Impact: Visitors looking for Houston permits via the state hub won't find it.
Fix: Either activate Houston's CITY_REGISTRY config, or make the state
     hub include cities from get_cities_with_data() (which includes all
     cities with permits, not just configured ones).

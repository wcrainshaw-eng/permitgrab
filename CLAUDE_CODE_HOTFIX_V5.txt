FIX 2: LEAD SCORING — FIXED-BREAKPOINT WITH DATASET AWARENESS  [MEDIUM]
===============================================================================

WHY V3 AND V4 BOTH FAILED:
- V3 used absolute thresholds. Result: 79-99 range (20-point spread).
  The problem was that ALL dashboard permits are recent, high-value, have
  contact info, and are active — so they all score high.
- V4 used percentile ranking on 50 returned permits. Result: 97-99 range
  (2-point spread). WORSE than V3. The problem was that percentile ranking
  on a pre-filtered dataset compresses to the top — if you rank 50 items
  that are already in the 80th+ percentile, they all get mapped to 96-100.

THE FIX — FORCED LINEAR SPREAD:
Do not use percentile ranking on the API results. Do not use absolute
thresholds. Instead, use a FORCED LINEAR SPREAD that maps the actual
min/max values in the dataset to the target range.

```python
def calculate_lead_scores(permits):
    """
    Score permits using forced linear spread across 40-99 range.
    Maps the actual data distribution to guarantee full range usage.
    """
    import hashlib
    from datetime import datetime

    if not permits:
        return []

    # Step 1: Calculate raw composite scores
    raw_scores = []
    for p in permits:
        raw = 0.0

        # Value component (0-35 points, linear within dataset)
        value = float(p.get('project_value') or p.get('estimated_cost') or 0)
        raw += value  # Store actual value; we'll normalize later

        raw_scores.append({
            'value': value,
            'filed_date': p.get('filed_date') or p.get('filing_date') or p.get('date'),
            'status': (p.get('status') or '').lower(),
            'has_phone': bool(p.get('phone') or p.get('contact_phone')),
            'has_name': bool(p.get('contact_name') or p.get('owner') or p.get('owner_name')),
            'trade': (p.get('trade') or p.get('permit_type') or p.get('trade_category') or '').lower(),
            'permit_id': str(p.get('id') or p.get('permit_number') or str(id(p))),
        })

    # Step 2: Find min/max for normalization
    values = [r['value'] for r in raw_scores]
    min_val = min(values) if values else 0
    max_val = max(values) if values else 1
    val_range = max_val - min_val if max_val > min_val else 1

    final_scores = []
    for i, r in enumerate(raw_scores):
        score = 0.0

        # Value: normalize to 0-35 within this dataset's range
        score += ((r['value'] - min_val) / val_range) * 35

        # Recency: 0-25 (days old)
        filed = r['filed_date']
        if filed:
            if isinstance(filed, str):
                try:
                    filed = datetime.strptime(filed[:10], '%Y-%m-%d')
                except:
                    filed = None
            if filed:
                days = (datetime.now() - filed).days
                score += max(0, 25 - (days * 0.8))  # Gradual decay

        # Status: 0-15
        status_map = {
            'filed': 15, 'active': 13, 'issued': 11, 'approved': 11,
            'permitted': 9, 'pending': 6, 'completed': 3, 'closed': 1
        }
        score += status_map.get(r['status'], 7)

        # Contact info: 0-10
        score += (5 if r['has_phone'] else 0) + (5 if r['has_name'] else 0)

        # Trade: 0-10
        high = ['electrical', 'hvac', 'plumbing', 'new construction', 'fire protection']
        mid = ['roofing', 'interior renovation', 'demolition', 'addition', 'structural']
        trade = r['trade']
        if any(t in trade for t in high):
            score += 10
        elif any(t in trade for t in mid):
            score += 7
        else:
            score += 4

        # Deterministic jitter from permit ID (0-4)
        jitter = int(hashlib.md5(r['permit_id'].encode()).hexdigest()[:4], 16) % 5
        score += jitter

        final_scores.append(score)

    # Step 3: FORCED LINEAR SPREAD — map actual score range to 40-99
    # This is the key difference from V3/V4. No matter what the raw scores
    # look like, we force them into the 40-99 output range.
    raw_min = min(final_scores)
    raw_max = max(final_scores)
    raw_range = raw_max - raw_min if raw_max > raw_min else 1

    result = []
    for s in final_scores:
        # Linear map: raw_min -> 40, raw_max -> 99
        normalized = 40 + ((s - raw_min) / raw_range) * 59
        result.append(max(40, min(99, int(normalized))))

    return result
```

KEY DIFFERENCE FROM V4:
The V4 percentile approach used sorted_scores.index(raw) / (n - 1), which
clusters when there are many tied values. This forced-spread approach uses
the actual continuous scores and linearly maps them, which preserves the
relative differences between permits.

The value component is normalized WITHIN the dataset (min/max), so even if
all permits are $500K-$2M, they get spread across the full 0-35 value
points. Combined with variation in recency, status, trade, and contact
info, the raw scores will have genuine variance that gets linearly mapped
to 40-99.

===============================================================================
COMMON MISTAKES TO AVOID (FROM V3 AND V4 FAILURES)
===============================================================================

1. DO NOT use file locking (fcntl) for duplicate prevention. It failed in V4.
   Use a DATABASE CONSTRAINT. That's what databases are for.

2. DO NOT apply percentile/ranking scoring to a pre-filtered result set.
   50 high-value permits will always cluster at 95-99. Use forced linear
   spread that maps min→40, max→99 regardless of input distribution.

3. DO NOT assume the application check alone prevents duplicates. Race
   conditions can bypass it. The database UNIQUE constraint is the safety
   net. Both layers are required.

4. DO NOT keep the NEWEST duplicate when cleaning up. Keep the OLDEST
   (MIN id) because it may have the Pro subscription and the user's
   original data.

5. DO NOT redirect non-Pro users to another page for feature gating.
   Use inline modals that preserve the user's current context.

6. ALWAYS verify on the LIVE site after deploying each fix. Do not rely
   on local testing alone — the production database may have different
   state (existing duplicates, different schema, etc).

7. READ the signup route COMPLETELY before modifying it. Print it to the
   terminal. Understand the full flow. The V3 fix was "added" but didn't
   work, which means it was likely in the wrong position or checking the
   wrong thing.

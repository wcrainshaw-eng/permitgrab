## KNOWN REMAINING RISKS (not blocking cutover attempt 3)

### LIKE case-sensitivity (V527+)
Postgres `LIKE` is case-sensitive; SQLite `LIKE` is ASCII-case-
insensitive by default. ~80 LIKE callsites in the repo are not
LOWER()-wrapped. On cutover, queries like `WHERE city LIKE 'chicago%'`
will not match rows where city is `Chicago, IL`. This is a correctness
drift, not a crash. Mitigation options for V527:
  - Wrap each LIKE in LOWER(...) at callsite.
  - Use ILIKE in the translator (`LIKE` → `ILIKE` only on Postgres path).
  - Use Postgres `citext` for the affected columns (schema change).

The translator could be extended to swap `LIKE` → `ILIKE` on the
USE_POSTGRES=true path. Decided NOT to ship in V525 because:
  - It changes semantics for SQLite's already-case-insensitive default
    (no-op), and changes Postgres behavior (now matches SQLite).
  - Some LIKE callsites may rely on case-sensitive behavior; switching
    to ILIKE for those would be a silent semantic shift.
  - Not blocking the cutover (correctness drift, not 503-crash).

### Daemon's collected_at TIMESTAMP comparison
The patched `INSERT INTO permits ... ON CONFLICT (permit_number) DO
UPDATE SET ... collected_at=excluded.collected_at` upserts work, but
freshness-window queries against `collected_at` use the V525 `(NOW() +
INTERVAL '-7 days')` translation. Cross-checked against Postgres
14.19 + psycopg2 2.9.11; column type is `TIMESTAMP DEFAULT NOW()` per
postgres_schema.sql. Comparisons match SQLite's behavior on the
`'YYYY-MM-DD HH:MM:SS'` text format SQLite stores.

### Date column TEXT vs Postgres date type
The schema's `filing_date` / `issued_date` / `date` /
`newest_permit_date` are `TEXT` in 'YYYY-MM-DD' form (SQLite legacy).
Postgres compares TEXT lexically — fine for ISO-8601 strings. The
`date('now', ...)` translation explicitly casts `(CURRENT_DATE +
INTERVAL '...')::date::text` so both sides of the `>=` are TEXT and
the lexical compare matches SQLite semantics.

## ARCHITECTURAL FOLLOW-UP (not for tonight)
Wes raised the SQLAlchemy question: should we be hand-rolling SQL
translation, or move to a proper ORM that handles dialects via a
backend abstraction layer? Pattern observation: every cutover attempt
finds the next layer of leakage (V522 = pool exhaustion, V525 = SQL
translation, future = NULL ordering / integer division / isoformat
representation / etc.). Hand-rolling means we ship one bug at a time
forever. SQLAlchemy Core would centralize this — but it's a sizable
migration. V527+ decision; not blocking the V522/V523/V525 cutover.
